# るびポン 読み補正ルール記述マニュアル

トークナイザが返す「読み」の誤りを補正するルール（ReadingRules）の仕組みと、
新しいルールを自分で書くための手順・実例をまとめたマニュアル。

**ルールの定義場所はただ1つ**: `furikana Extension/Resources/reading-rules.js`
（Safari 版・Chrome 版で共有。ここだけ編集すれば両方に反映される）

> 💡 **先にユーザー辞書を検討する**: 「この単語は常にこの読み」という単純な補正
> （例: 五月雨→さみだれ）は、コードを書かずに設定画面のユーザー辞書（TSV）で
> 実現できる。ルールをコードに書くのは、①全ユーザーに配布したい補正、
> ②数字パターンや前後文脈が絡む補正、の場合。

---

## 1. 内部の仕組み

### 1.1 パイプラインの中でルールが動く位置

```
Webページのテキストノード
  → トークナイザ（3種のいずれか）
      ├─ システム辞書: Swift (NLTagger + CFStringTokenizer)
      ├─ IPA辞書:     kuromoji.js
      └─ Sudachi辞書:  Sudachi WASM
  → content.js が結果を受信
      → splitParenReadingTokens()   … 括弧付き読み注釈の分割
      → ★ ReadingRules.apply()      … ここで読み補正（このマニュアルの対象）
      → groupToLongUnits()          … 長単位へのグルーピング
      → splitKanjiReading()         … 送り仮名の分離
      → <ruby> 要素を構築して DOM へ
```

重要な帰結:

- **ルールはどの辞書バックエンドでも同じように適用される**（JS側に一元化）
- ルールが受け取るのは「トークンの配列」であり、ページの生テキストではない
- ルールの後段で長単位化・送り仮名分離が行われるため、ルールは
  「トークンの読みを正しくする」ことだけに集中すればよい
- 設定「読み補正ルールを適用」が OFF のときは `apply()` 自体が呼ばれない

### 1.2 トークンとは

すべての辞書バックエンドは共通形式のトークンを返す:

```js
{ surface: '東京', reading: 'とうきょう', pos: 'Noun', range: [0, 2] }
```

| フィールド | 内容 | ルール記述での扱い |
|---|---|---|
| `surface` | 表層形（ページに書かれている文字列） | **マッチングの主役**。すべてのルールは surface を見る |
| `reading` | 読み（常にひらがな） | ルールが**上書きする対象** |
| `pos` | 品詞（英語タグ） | **当てにしない**。システム辞書では全トークン `OtherWord` になる（NLTagger が日本語の品詞タグ非対応のため）。ルールの条件にも使えない |
| `range` | 元テキスト中の位置 `[開始, 終了]`（UTF-16） | ルールでは触らない（sequence ルールの統合時に自動計算される） |

### 1.3 apply() の処理順序

`ReadingRules.apply(tokens)` は 2 段階で動く:

```
1. applySequenceRules(tokens)
     隣接する複数トークンを1つに統合して読みを与える（例: 何|を → 何を=なにを）
     左から右へ1パス。各位置で priority の高いルールから順に試し、
     最初にマッチしたルールがトークンを消費する

2. applySimpleRules(tokens)
     各トークンについて:
       a. surface ルール（表層形の完全一致）を priority 順に試す → 最初のマッチで確定
       b. マッチしなければ regex ルールを priority 順に試す → 最初のマッチで確定
```

つまり **sequence → surface → regex** の順。sequence で統合された新しいトークン
（surface が連結される）に対しても、続く surface / regex ルールは適用される。

### 1.4 優先度（priority）とユーザー辞書の合流

- ルールは初期化時に priority **降順**でソートされる（毎回ソートしない
  キャッシュ方式。ファイル内の記述順は同 priority 内の順序にのみ影響）
- ユーザー辞書（TSV）から生成されたルールは priority 200（surface）/
  210（sequence）で**組み込みルールより常に優先**される
- 既存ルールの priority 相場:

| 帯 | 用途 |
|---|---|
| 90〜95 | surface ルール（95 = より長い語を先に） |
| 100〜110 | sequence ルール（110 = より分割が細かいバリエーション） |
| 129〜136 | regex ルール（**特殊形 130〜136 > 一般形 129** の慣例。後述） |
| 200 / 210 | ユーザー辞書（予約済み。組み込みルールでは使わない） |

### 1.5 数字最適化（知っておくと事故を防げる）

regex ルールは「パターンに数字を含むルール」と「含まないルール」に事前分類され、
**トークンの surface に数字（半角・全角）が無い場合、数字系ルールはスキップ**される
（`HAS_DIGIT = /[0-9０-９]/`）。つまり数字絡みのルールを書くときは、パターンの
文字クラスに**半角と全角の両方**（`[0-9０-９]`）を入れておけばよく、
数字を含まない語への性能影響を心配する必要はない。

### 1.6 発火ログ

ルールが読みを変更すると Web インスペクタのコンソールに出力される:

```
[Furikana] ReadingRule applied: "番" ふぁあん → ばん
[Furikana] ReadingRule merged: 11 → 9 tokens   ← sequence 統合が起きたとき
```

「ルールを書いたのに効かない」ときは、まずこのログの有無を確認する。

## 2. ルールを書く前に知っておくべきこと

### 2.1 3種類のルールと使い分け

| 種類 | マッチ対象 | 典型用途 | 例 |
|---|---|---|---|
| **sequence** | 隣接する複数トークンの surface 列 | 辞書が語を分割してしまう場合の統合＋読み | `['何','を']` → なにを |
| **surface** | 1トークンの surface 完全一致 | 特定の語の読み違い | `令和` → れいわ |
| **regex** | 1トークンの surface への正規表現 | 数字+助数詞、パターン化できる語 | `/^([0-9０-９]+)番$/` → `$1ばん` |

### 2.2 最重要: 辞書によってトークン分割が違う

同じ「7月」でも、バックエンドによってトークンの切れ方が異なる:

| バックエンド | 「7月」の分割 | 傾向 |
|---|---|---|
| システム辞書 | `7` \| `月` | 数字と助数詞が分かれる |
| IPA辞書 (kuromoji) | `7` \| `月` | 細かく分割 |
| Sudachi | `7月`（結合されることが多い） | 長単位 |

このため、**数字系・複合語系のルールは「結合形」と「分割形」をペアで書く**のが
このプロジェクトの基本パターンである:

- **combined ルール**: `7月` のような 1 トークンに regex でマッチ
- **split ルール**: `月` 単独トークンに、`context.prevPattern` で「直前が数字」を条件にマッチ

語の統合系も同様に、分割パターン全部に備えて **sequence（分割形）と
surface（結合形）をペアで書く**（実例は §4.1）。

### 2.3 フィールド仕様

**sequence ルール**:
```js
{ id: 'nani-wo-seq',       // 一意なID（慣例: 意味-seq / -surface / -combined / -split）
  priority: 100,
  surfaces: ['何', 'を'],   // ★複数形 surfaces（配列）
  reading: 'なにを',
  context: { ... } }        // 省略可
```

**surface ルール**:
```js
{ id: 'rei-wa-surface',
  priority: 90,
  surface: '令和',           // ★単数形 surface（文字列）
  reading: 'れいわ' }
```

> ⚠️ **最頻出の罠**: sequence は `surfaces`（複数形・配列）、surface は
> `surface`（単数形・文字列）。取り違えても**エラーは出ず、無言でマッチしなく
> なる**。効かないときは真っ先にここを疑う。

**regex ルール**:
```js
{ id: 'ban-combined',
  priority: 130,
  pattern: /^([0-9０-９]+)番$/,   // surface 全体にマッチさせる（^ $ を必ず付ける）
  replacement: '$1ばん',          // ★これが「新しい読み」になる
  context: { ... } }              // 省略可
```

> ⚠️ **regex ルールの本質**: 新しい読みは
> `surface.replace(pattern, replacement)` で作られる。つまり
> **replacement は「表層形から読みを生成する式」**である。
> `$1` 等のキャプチャで数字部分を読みに残せる（`15番` → `15ばん`。
> 数字部分のルビは後段の処理で自動的に省略されるので気にしなくてよい）。
> replacement に**漢字を残してはいけない**（読みにならない）。

**context（sequence / regex 共通、すべて省略可）**:
```js
context: {
    prevPattern:     /^[0-9０-９]+$/,  // 直前トークンの surface に対する正規表現
    prevPrevPattern: /^月$/,           // 2つ前のトークンの surface
    nextPattern:     /^で$/,           // 直後のトークンの surface
}
```

- 判定対象は**隣接トークンの surface 全体**（文字ではない）。
  例えば `prevPattern: /[0-9０-９]$/`（アンカーなし）は「直前トークンが数字で
  **終わる**」の意味になり、`15` にも `第15` にもマッチする —
  これを利用して**末尾桁による音便判定**ができる（§4.4）
- 文書の先頭・末尾では隣接トークンは空文字 `''` として扱われる
- sequence ルールの `nextPattern` は「統合対象の列の直後」を見る

## 3. 記述手順（ステップバイステップ）

### Step 1: 誤読の実例を採取する

「どのページの・どの語が・何と読まれ・正しくは何か」を明確にする。
例（実際にあった報告）: システム辞書で「1番」が「1**ふぁあん**」になる。
正しくは「1ばん」。

### Step 2: トークンの切れ方を確認する

ルールは surface に対して書くので、対象の語が**どう分割されているか**を知る必要がある。

方法A — 実機のコンソールログ: Web インスペクタで
`[Furikana] Tokenization successful` 前後のログを見る。

方法B — Node ハーネス（後述 §3.5 のスニペットに、想定トークンを手で書いて確認）。

方法C — 迷ったら**結合形と分割形の両方に備えたペアで書く**（§2.2）。
これが一番確実で、既存ルールもすべてこの方針。

今回の例: システム辞書は `1`|`番` と分割 → 分割形が必要。
Sudachi は `1番` と結合し得る → 結合形も書く。

### Step 3: ルール種類を選ぶ

```
補正したいのは…
├─ 固定の1語（分割されない） ………………………………… surface ルール
├─ 固定の語だが辞書によって分割される ……………………… surface + sequence のペア
└─ 数字や可変部分を含むパターン ……………………………… regex ルール
     └─ 辞書によって分割される …………… combined + split のペア（context 使用）
```

### Step 4: reading-rules.js の該当配列に追記する

ファイル内の構成（配列は3つ）:

```
reading-rules.js
├─ const sequenceRules = [ ... ]   ← sequence ルールはここ
├─ const surfaceRules  = [ ... ]   ← surface ルールはここ
└─ const regexRules    = [ ... ]   ← regex ルールはここ（助数詞ごとにセクション化されている）
```

似た既存ルールの近く（例: 助数詞なら他の助数詞セクションの隣）に、
セクションコメント付きで追加する。

### Step 5: Node で単体検証する

ビルド不要で、その場でルールの発火を確認できる:

```bash
cd /path/to/furikana
node - <<'EOF'
const fs = require('fs');
const src = fs.readFileSync('furikana Extension/Resources/reading-rules.js', 'utf8');
eval(src + '; globalThis.RR = ReadingRules;');

// Step 2 で確認した分割どおりにトークンを書く（誤った読みを入れる）
const tokens = [
    { surface: '15', reading: '15', pos: 'Noun', range: [0, 2] },
    { surface: '番',  reading: 'ふぁあん', pos: 'Noun', range: [2, 3] },
];
console.log(JSON.stringify(RR.apply(tokens), null, 2));
EOF
```

期待する読みに変わること、および**誤発火しないべきケース**
（例: 「交番」「当番」）が変化しないことの両方を確認する。

### Step 6: 構文チェック → ビルド → 実機確認

```bash
node -c "furikana Extension/Resources/reading-rules.js"   # 構文チェック
npm run build:chrome                                       # Chrome版: dist/chrome を再生成 → 拡張を再読み込み
# Safari版: Xcode でビルド（JSのみの変更でも appex に再コピーが必要）
```

実ページで対象の語を表示し、コンソールに
`[Furikana] ReadingRule applied:` が出ることを確認する。

## 4. 実例で学ぶ

### 4.1 実例: surface + sequence のペア —「令和」

「令和」は辞書によって `令和`（1トークン）とも `令`|`和`（2トークン）とも
切られる。両方に備えて2本書く:

```js
// sequenceRules 側（分割形: 令|和 を統合して読みを与える）
{ id: 'rei-wa', priority: 100, surfaces: ['令', '和'], reading: 'れいわ' },

// surfaceRules 側（結合形: 令和 1トークンの読みを上書き）
{ id: 'rei-wa-surface', priority: 90, surface: '令和', reading: 'れいわ' },
```

sequence が先に走るので、分割形はまず統合され（surface が「令和」になる）、
その時点で読みも確定する。surface ルールは結合形で来た場合の保険。

### 4.2 実例: 分割バリエーションを priority で捌く —「何曜日」

「何曜日」は 4 通りの分割があり得る。**より細かい分割（トークン数が多い）ルールに
高い priority** を与え、長いマッチを先に取らせる:

```js
{ id: 'nan-youbi-seq-3',  priority: 110, surfaces: ['何', '曜', '日'], reading: 'なんようび' },
{ id: 'nan-youbi-seq-2a', priority: 110, surfaces: ['何曜', '日'],     reading: 'なんようび' },
{ id: 'nan-youbi-seq-2b', priority: 110, surfaces: ['何', '曜日'],     reading: 'なんようび' },
{ id: 'nan-youbi-surface', priority: 95, surface: '何曜日',            reading: 'なんようび' },
```

より一般的な `['曜','日']=ようび`（priority 100）や `曜日`（90）より高くすることで、
「何」を含む長い形が先にマッチし、「なんようび」が「なん＋ようび」に
分解されてしまうのを防いでいる。

### 4.3 実例: regex の combined + split ペア —「数字 + 番 = ばん」

実際にあった不具合（システム辞書で「1番」→「1ふぁあん」）への対処として
追加されたルール。新規追加の手順をなぞる:

**Step 1-2**: 誤読はシステム辞書の `番` 単独トークン。分割は `1`|`番`。
Sudachi では `1番` 結合もあり得る。「番」は音便変化がない（1番=いちばん、
3番=さんばん）ので読みは常に「ばん」。

**Step 3**: 数字+可変 → regex。分割に備えて combined + split のペア。

**Step 4**: regexRules の助数詞セクションに追加:

```js
// === 番: 数字の後は常に「ばん」（1番=いちばん、音便変化なし）===
{ id: 'ban-combined', priority: 130,
  pattern: /^([0-9０-９]+)番$/,     // 「15番」1トークンに ^$ 付きでマッチ
  replacement: '$1ばん' },          // 読み = 「15ばん」（数字は読みに残す）
{ id: 'ban-split', priority: 130,
  pattern: /^番$/,                  // 「番」単独トークン
  replacement: 'ばん',
  context: { prevPattern: /[0-9０-９]$/ } },  // 直前トークンが数字で終わる
```

ポイント:
- combined の replacement は `$1` で数字を読みへ引き継ぐ（表層 `15番` → 読み `15ばん`）
- split の `prevPattern` にアンカー `^` を付けていないのは意図的で、
  「第15」のように数字で**終わる**トークンにもマッチさせるため
- 「交番」「当番」は surface が `番` 単独でない・直前が数字でないため誤発火しない

**Step 5** の検証はこのマニュアル §3.5 のスニペットそのまま
（`ふぁあん` が `ばん` に変わり、`当|番` が変化しないことを確認済み）。

### 4.4 実例: 音便（読みの変化）を桁で判定する —「軒」

助数詞には数字によって読みが変わるものがある。「軒」は「3軒=さん**げん**」
だけが濁り、他は「けん」。**特殊形に高い priority（130）、一般形に低い
priority（129）** を与えて、特殊が先にマッチするようにする:

```js
// === 軒: げん(単独3のみ) / けん(その他) ===
{ id: 'gen-combined', priority: 130,
  pattern: /^([3３])軒$/,          // 「3軒」だけ
  replacement: '$1げん' },
{ id: 'ken-combined', priority: 129,
  pattern: /^([0-9０-９]+)軒$/,    // その他すべての「N軒」
  replacement: '$1けん' },
{ id: 'gen-split', priority: 130,
  pattern: /^軒$/,
  replacement: 'げん',
  context: { prevPattern: /^[3３]$/ } },      // ^ あり = 「3」単独のみ（13軒は対象外）
{ id: 'ken-split', priority: 129,
  pattern: /^軒$/,
  replacement: 'けん',
  context: { prevPattern: /[0-9０-９]$/ } },  // ^ なし = 数字で終わればすべて
```

このプロジェクトの音便方針（CLAUDE.md にも記載）:
- **階/軒/百/千**: 濁音・半濁音化は「単独桁の3」のみ（3階=さんがい、13階=じゅうさんかい）
  → `prevPattern: /^[3３]$/`（アンカーあり）
- **杯/本/匹/泊/分**: 末尾桁で判定（13杯=じゅうさん**ばい**）
  → `prevPattern: /[3３]$/`（アンカーなし＝末尾桁を見る）

### 4.5 実例: 2つ前のトークンを見る —「ヶ月」の分割形

「1ヶ月」が `1`|`ヶ`|`月` と3分割された場合、「月」の読みを「げつ」にするには
直前（ヶ）だけでなく**2つ前（数字）**の確認が要る:

```js
{ id: 'kagetsu-split', priority: 135,
  pattern: /^月$/,
  replacement: 'げつ',
  context: {
      prevPattern: /^[かヶケヵカ]$/,          // 直前は「ヶ」系1文字
      prevPrevPattern: new RegExp('^(' + ANY_NUM + ')$')  // 2つ前は数字
  } },
```

`new RegExp('...' + 変数 + '...')` の形は、数字クラスなどの共通部品
（`ANY_NUM`, `PAKU_LAST` 等、ファイル冒頭付近で定義）を再利用するときに使う。

## 5. 落とし穴チェックリスト

書き終えたら以下を確認する:

- [ ] **surfaces / surface を取り違えていないか**（sequence=複数形、surface=単数形。間違えると無言で不発）
- [ ] regex の pattern に **`^` と `$`** を付けたか（付けないと「交番」の「番」にも部分マッチする）
- [ ] 数字クラスに**全角**を含めたか（`[0-9０-９]`。ニュースサイトは全角数字が多い）
- [ ] `prevPattern` のアンカーは意図どおりか（`^N$`=単独桁のみ / `N$`=末尾桁判定）
- [ ] replacement は**ひらがな＋数字のみ**か（漢字を残すと読みが壊れる）
- [ ] 同じ surface にマッチする既存ルールと **priority が競合**していないか（特殊 > 一般）
- [ ] **誤発火しないべき語**（同じ字を含む別語）で Node 検証したか
- [ ] `node -c` で構文チェックしたか
- [ ] `id` が既存と重複していないか（動作には影響しないがデバッグで混乱する）

## 6. デバッグ

### ルールが効かないとき（頻度順）

1. `surfaces`/`surface` の取り違え（§2.3 の罠）
2. トークン分割が想定と違う → 実際の分割を確認（§3 Step 2）。
   結合形/分割形のもう片方を書き足す
3. より高い priority の別ルールが先にマッチしている →
   コンソールの `ReadingRule applied` ログでどのルールが動いたか推定し、
   priority を調整
4. 設定「読み補正ルールを適用」が OFF / 拡張のリロード漏れ
   （Chrome は `npm run build:chrome` + 拡張再読み込み、Safari は Xcode 再ビルドが必要）
5. ユーザー辞書（priority 200/210）が同じ surface を先取りしている

### 確認場所

- **発火ログ**: 対象ページの Web インスペクタ（Safari: 開発メニュー / Chrome: DevTools）
  で `[Furikana] ReadingRule` をフィルタ
- **単体検証**: §3.5 の Node ハーネス（ビルド不要・最速）
- ルールより前の段階（トークナイザ自体の問題）を疑う場合:
  `docs/tokenization-architecture.md` を参照

## 付録: 用語対応表

| 用語 | 意味 |
|---|---|
| 表層形（surface） | ページに実際に書かれている文字列 |
| 読み（reading） | ふりがなとして表示されるひらがな |
| 結合形 / combined | 数字と助数詞などが1トークンになっている状態（例: `15番`） |
| 分割形 / split | 別々のトークンに分かれている状態（例: `15`\|`番`） |
| 音便 | 数字に応じた読みの変化（連濁・半濁音化・促音化。例: 3杯=さんばい） |
| 長単位化 | ルール適用後、表示のためにトークンをまとめる処理（ルールとは無関係） |
